Integration
Sync & Integration

How the CRM connects to the Command Centre and Management Hub in real time — across all team members' devices.

1
Team member saves an opportunity, lead, or activity in the CRM
T+0s
↓ debounce 1.5s (groups rapid changes)
2
Sync engine computes aggregates: pipeline by initiative, proposal counts, conversion rates, owner breakdowns, conference stats
T+1.5s
↓ HTTP POST to Supabase REST API
3
Supabase writes to crm_sync table — row "pipeline_sync" updated, row "hub_summary" updated
T+1.6s
↓ Postgres change event → WebSocket push
4
Command Centre receives realtime push · applyCRMOverlay() runs · Pipeline, Proposals, Reconciliation, Meeting Health sections re-render
T+2s
↓ parallel
5
Management Hub reads hub_summary on next load — pipeline strip updates
Next load
📡
Sync Status Indicator
The sync dot in the CRM topnav shows the current sync state at all times.
🟡 Pulsing = syncing to Supabase now
🟢 "Synced HH:MM" = confirmed written to shared database
🔴 "Offline — local only" = Supabase unreachable, saved locally
Click the indicator to force a manual sync at any time
🛡️
Offline Fallback
If Supabase is unreachable, every save still writes to localStorage. No data is lost. On reconnection, the next save syncs everything.
localStorage is the primary data store for each user's CRM records
Supabase holds only aggregated pipeline data (no individual deal records)
CC falls back to localStorage copy if Supabase unavailable
ℹ️
What Supabase stores vs what stays local: Individual deal names, company names, contact details, and notes stay in each user's browser localStorage only. Supabase receives only aggregated values — total pipeline in $M per initiative, counts of open/stalled deals, won MTD. No PII or deal-level detail is in the cloud database.
Operating Rules
Built-in Commercial Rules

Rules from the BU Services Commercial Operating Model are enforced directly in the CRM UI — not documented separately, built in.

Pipeline Management
30-day stale rule — any open opportunity with no logged activity in 30+ days shows ⚠ Stale flag. Banner count shown at top of pipeline.
Friday Salesforce update — the CRM auto-syncs to Supabase on every save. The equivalent of the mandatory Friday SF update is implicit: when you save, you sync.
Stage discipline — five stages map directly to BU Services' commercial process. Closed Lost is not deleted — it stays for conversion rate calculation.
Close date tracking — past close dates on open deals highlight in red. Deals with close dates within 14 days highlight in amber.
Conference Rules
48-hour lead entry — all conference leads must be entered within 48 hours of returning. The CRM shows a ⏰ deadline badge and a "Needs Entry" filter.
Lead target — each conference record has a target leads count. Progress bar goes red if entered leads fall below target.
Lead source tracking — every lead has a source field with all 2026 BU Services conferences pre-loaded as options.
3X Composite Rule
AMBER/RED finding flag — when logging an activity, any field inspection finding can be flagged as AMBER or RED. This triggers a toast alert to create a 3X Composite opportunity.
Visual flag — flagged activities show a coloured badge in the activity feed so nothing is missed in review.
Syncs to CC — 3X pipeline total flows to the Command Centre initiative tracker.
Data Integrity
Required fields — opportunity name, company, initiative, stage, amount, close date, and owner are required. The form blocks save if any are missing.
Deletion confirmation — all five record types (opportunities, leads, accounts, activities, conferences) require a confirmation prompt before deletion. Cannot be undone.
Auto-account creation — saving a new company name automatically creates an Account record. Prevents orphaned opportunities.
Data & Storage
Data Storage & Export

Where your data lives, how to back it up, and what to do if you need to move devices.

💾
localStorage — Individual Records
Every deal, lead, account, activity, and conference you enter is stored in your browser's localStorage under five keys.
bucrm_opps — all opportunity records
bucrm_leads — all lead records
bucrm_accounts — all account records
bucrm_activities — all activity records
bucrm_conferences — all conference records
🗄️
Supabase — Aggregates Only
Supabase stores only two rows of computed aggregate data — no individual deal details. Safe to leave the anon key in the HTML files.
pipeline_sync — full aggregate payload ($M totals, counts, owner breakdown)
hub_summary — lightweight summary for Management Hub strip
No PII, no deal names, no company names in Supabase
↓
JSON Export
Export all CRM data to a structured JSON file at any time. Dashboard view → Export JSON button (top right).
Filename: BU_Services_CRM_YYYY-MM-DD.json
Contains all five record types in full
Use for backup before clearing browser data
Use for migration to new device (import not yet built — copy-paste seedDemoData pattern)
🔄
Multi-Device / Multi-User
Individual deal records are per-browser. Aggregated pipeline data is shared via Supabase. Each commercial team member runs their own CRM locally.
Emmanuel's deals → his localStorage → Supabase → Jon's CC
Rick's deals → his localStorage → Supabase → Jon's CC
All team members' pipeline aggregates converge in CC in real time
Full team deal list requires Phase 2 (Graph API + SharePoint backend)
⚠️
Important: Clearing your browser data (cookies & site data) will delete all CRM records from localStorage. Export JSON regularly as your backup. The Supabase aggregates will survive a browser clear but your individual deal records will not.
